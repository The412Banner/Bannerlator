What is VEGAS?

VEGAS is a community performance fork of GPLAsync 2.7.1 (itself a DXVK fork with async pipeline compilation) specifically tuned for Qualcomm Adreno GPUs on Android. It was originally developed as part of the Star Engine / Star Emulator project and later released as a standalone GPLAsync/DXVK fork.

VEGAS adds a tier-based auto-tuning engine, FSR 1.0 compute upscaling with async dispatch, motion-compensated frame generation, a TBDR-aware adaptive draw governor, GPU pacing (HAAE), pipeline bind-skip optimization, per-tier shader zero-init, and more — all behind a single master switch.

How is VEGAS different from stock DXVK?

Stock DXVK is designed for desktop GPUs (NVIDIA, AMD, Intel) with an immediate-mode renderer. It assumes a desktop GPU architecture where more draw calls = better GPU utilization.

Adreno GPUs are Tile-Based Deferred Renderers (TBDR). On TBDR, accumulating too many draw calls before flushing the tile buffer causes catastrophic performance collapse (tile spill to system RAM). VEGAS addresses this with:

  • TBDR-inverted governor — lowers the draw threshold when CPU-bound (opposite of desktop DXVK)
  • Adaptive draw flushing — spills render passes before tile overflow
  • GPU pacing (HAAE) — injects empty submits to prevent queue buildup
  • Tier system — auto-tunes thresholds per GPU capability
  • Bind-skip — reduces redundant vkCmdBindPipeline calls
  • Compiler thread cap — limits shader compiler threads to 4 on ARM64 (prevents CPU contention on big.LITTLE SoCs)
Does VEGAS have async pipeline compilation like the async-arm64ec builds of DXVK?

Yes — and more. VEGAS supports async pipeline compilation via the standard dxvk.enableAsync = True option. When enabled, shader compilation is offloaded to worker threads so the game never waits for a pipeline to finish compiling before it can render a frame. This is the same mechanism used by the async-arm64ec builds.

However, VEGAS goes further than plain async DXVK in three ways:

  1. GPL + async compatibility fix — Stock DXVK's Graphics Pipeline Libraries (VK_EXT_graphics_pipeline_library) and async compilation are mutually incompatible. When both are active, GPL eagerly compiles pipelines that async is trying to defer, causing race conditions and driver crashes on Turnip. VEGAS automatically disables GPL when async is enabled (canUseGraphicsPipelineLibrary() returns false when dxvk.enableAsync is true). This is the single biggest stability fix for async on Adreno — without it, async builds crash on many games.
  2. Compiler thread cap — On ARM64, VEGAS caps auto-detected compiler threads to 4 (instead of 8 on a typical big.LITTLE SoC). This prevents the shader compiler threads from competing with the game's main and render threads for CPU time, which reduces frame-time spikes during compilation. You can override with dxvk.numCompilerThreads = N.
  3. 60-second shader cache flush — The shader cache writer thread uses a 60-second timeout instead of an indefinite wait. This ensures partially-filled cache batches are written to disk even if the emulator is killed (SIGKILL). Without this, cached shaders from the previous session could be lost, forcing recompilation and stutter all over again on the next launch.
Enable async: dxvk.enableAsync = True. Combined with VEGAS's GPL fix and thread cap, this gives you the smoothest shader compilation experience available on Adreno. See FAQ #16 if you still experience stutters.

How do I install VEGAS?

VEGAS ships as a WCP package for use with Winlator or Star Emulator:

  1. Download the latest .wcp from the releases page.
  2. In Star Emulator: open Contents menu → install the vegas-*.wcp package.
  3. In Winlator: install the dxvk-*-vegas-*.wcp package as a standard DXVK WCP.

For manual install, extract the DLLs (d3d9.dll, d3d11.dll, dxgi.dll, etc.) to the Wine prefix's system32/ and syswow64/ directories.

Check the release assets: each release provides two WCPs — one with type: "DXVK" (for Winlator) and one with type: "VEGAS" (for Star Emulator).
Where does VEGAS read its config from?

VEGAS reads configuration from a dxvk.conf file. The search order is:

  1. DXVK_CONFIG_FILE environment variable (full path to a config file)
  2. /storage/emulated/0/Winlator/dxvk.conf
  3. /storage/emulated/0/Download/dxvk.conf
  4. /storage/emulated/0/dxvk.conf

The recommended config with all VEGAS options documented is included in each release's vegas-config artifact. You can copy it to one of the paths above and edit as needed.

Alternatively, set options via environment variable: DXVK_CONFIG="dxvk.enableStarProfile=True vegas.enableUpscaler=True"
What does the master switch do?

The master switch is dxvk.enableStarProfile (Tristate: Auto / True / False):

  • Auto (default) — Enables all VEGAS features automatically when an Adreno GPU is detected. Disables everything on non-Adreno GPUs.
  • True — Force-enables all VEGAS features regardless of GPU.
  • False — Hard-disables every VEGAS feature. The DLL behaves like stock DXVK. Use this as an emergency escape if a game has issues with VEGAS.

Example: dxvk.enableStarProfile = True

How do I configure FSR?

FSR 1.0 upscaling is controlled by vegas.enableUpscaler (Tristate: Auto / True / False):

  • Auto (default) — Upscales only when the game's render resolution is lower than the swapchain (display) resolution.
  • True — Always upscale. Forces FSR to run every frame regardless of resolution.
  • False — FSR disabled.

Full example config block:

vegas.enableUpscaler = True
dxvk.enableStarProfile = Auto   # master switch must not be False

FSR 1.0 is an edge-sharpening (spatial) upscaler, not temporal like FSR 2.0+. It sharpens edges when upscaling from a lower resolution. On mobile screens the effect can be subtle.

FSR doesn't seem to do anything — why?

This is the most common question. Here are the probable reasons:

  1. Auto mode + native resolution — If the game renders at your display's native resolution (e.g., 1080p on a 1080p display), Auto mode considers upscaling unnecessary and never dispatches FSR. You must either lower the game's internal resolution or set vegas.enableUpscaler = True.
  2. FSR 1.0 is visually subtle — Unlike FSR 2/3 which reconstructs detail temporally, FSR 1.0 is purely a spatial edge sharpener. On a 6" phone screen at arm's length, the difference between native 540p and FSR-upscaled-to-1080p is often imperceptible unless you freeze-frame and pixel-peep.
  3. The game's resolution setting may be ignored — Some games in Winlator/Star Emulator advertise a resolution option but still create a backbuffer at native resolution. Check with DXVK_HUD=devinfo to see the actual render resolution.
  4. Async dispatch can drop frames — FSR dispatches asynchronously via timeline semaphore. If the GPU is too busy, the compute shader may not finish before the next present, and VEGAS skips the blit entirely — no error, just no FSR on that frame.
  5. Config not being read — Ensure your dxvk.conf is in one of the correct paths (see FAQ #6) and that the syntax is correct. Set dxvk.enableStarProfile = False as a test — if other VEGAS features (like the colored frametime graph) stop working, the config is being read.
  6. Swapchain format guard — FSR only runs on UNORM swapchain formats. Some games use sRGB or other formats. This is a safety check to prevent crashes on Turnip.
Quick test: Set vegas.enableUpscaler = True, set the game to its lowest possible resolution, and watch adb logcat -s DXVK for lines containing "Upscale" or "fsr". If you see them, FSR is running — you just can't see it.
How do I verify FSR is actually running?

Use these methods:

  1. Logcat — Run adb logcat -s DXVK | grep -iE "upscale|fsr|vegas:". VEGAS logs performance state every frame (Vegas: Perf=... fsrActive=yes).
  2. HUD — Set DXVK_HUD=frametimes,fps,gpuload,devinfo. The GPU load percentage shifts when FSR is active (it adds compute work, then reduces per-pixel shading load).
  3. Frame time analysis — FSR should reduce GPU frame time when upscaling from a lower resolution, but add ~0.1ms of compute dispatch overhead.
  4. Visual freeze-frame — Set the game to exactly half resolution (e.g., 540p on 1080p). Take a screenshot with FSR on and off. Compare edge sharpness at 2× zoom. FSR 1.0's effect is visible in edge aliasing reduction.
The async design means FSR output from frame N appears on frame N+1. This is normal and not a bug.
What is the Tier system?

VEGAS classifies Adreno GPUs into 3 tiers (1=entry, 2=mid, 3=high-end) using the GPU model name from the KGSL sysfs node. Each tier receives automatically tuned parameters:

ParameterTier 1Tier 2Tier 3
Draw Threshold (base)100200350
HAAE Pacing Threshold50100150
Governor Cap Multiplier2.0×2.0×1.7×
Shader Zero-InitONONOFF

GPU PersonaGTX 1050 TiGTX 1070RTX 3060

You can override the detected tier with vegas.forceTier = 1/2/3. Set to 0 for auto-detection (default).

D3D9 games get higher base thresholds ({300, 500, 800}) because they typically issue more draw calls per frame. The TBDR-aware governor handles the rest.
What is the Adaptive Governor?

The Adaptive Governor is VEGAS's key TBDR optimization. It dynamically adjusts the draw call threshold (how many draws accumulate before a render pass is flushed) based on real-time frame time analysis:

  • GPU-bound (load > 0.90, frameTime > 25ms) → RAISES the threshold — batch more draws to amortize submission overhead.
  • CPU-bound (load < 0.40, frameTime > 12ms) → LOWERS the threshold — flush earlier to prevent tile-buffer starvation on TBDR.
  • Balanced → resets to base threshold.

Why inverted? Desktop DXVK raises the threshold for both CPU-bound and GPU-bound scenarios. On TBDR Adreno, raising the threshold when CPU-bound makes the problem worse — more draws accumulate in the tile buffer, increasing driver overhead and starving the GPU. VEGAS correctly reduces the threshold when CPU-bound.

The governor uses EMA smoothing (0.9×prev + 0.1×current frameTime) and a 15-frame cooldown (~250ms at 60fps) to prevent oscillation.

What is Frame Generation?

VEGAS includes a 3-pass motion-compensated frame generation pipeline that runs entirely on the GPU (compute shaders). It synthesizes an interpolated frame between each pair of real frames:

  1. Motion Estimation (16×16 tiles) — Block Sum of Absolute Differences between previous and current frame → raw motion vectors.
  2. Median Filter (8×8 tiles) — 3×3 spatial denoise on the motion field to remove outliers.
  3. Warp + Blend (8×8 tiles) — Warp the previous frame by the filtered motion vectors, alpha-blend at 0.5 weight.

Eligibility:

  • Tier 1: Disabled (insufficient compute budget)
  • Tier 2: frameTime ≤ 29ms (≥34 FPS headroom)
  • Tier 3: frameTime ≤ 33ms (≥30 FPS headroom)
Frame generation increases perceived smoothness but adds ~3-5ms of GPU compute time per frame. It is not "free FPS" — use it only when you have performance headroom above 30 FPS.

How do I enable performance logging?

Enable the DXVK log via environment variable or container configuration:

DXVK_LOG_LEVEL=info    # Normal logging
DXVK_LOG_LEVEL=debug   # Verbose (includes per-frame Vegas metrics)

Then watch the output:

adb logcat -s DXVK | grep "Vegas:"

The per-frame log line looks like:

Vegas: Perf=LAGGING ftRatio=1.5 load=0.92 frameTime=25.0ms fsrActive=yes frameGen=no

This shows the performance state, frame time ratio, GPU load estimate, and whether FSR/frame generation are active on the current frame.

Why do I still get shader stutters? I heard VEGAS eliminates them by 90%

This is the most important troubleshooting question. Let's be clear: VEGAS dramatically reduces shader compilation stutter compared to stock DXVK on Adreno, but it cannot eliminate it entirely. Here is exactly why stutters can still happen and what to do about each cause:

1. Async pipeline compilation is not enabled (most common)

If you didn't set dxvk.enableAsync = True, you're running with synchronous compilation. Every time the game encounters a new shader, the render thread blocks until it's compiled. This is the #1 cause of stutter. Fix:

dxvk.enableAsync = True

Without this, VEGAS's GPL fix, thread cap, and cache flush still help, but the game still waits for compilation on the main thread.

2. The GPL-Async race still triggers on some Turnip builds

VEGAS disables GPL when async is enabled, but some Mesa Turnip versions have their own bugs with async pipeline creation. If you still get stutter or crashes with async enabled, try an older or newer Turnip build (Mesa 24.3.x vs 25.x). The Turnip async path has improved significantly across Mesa versions.

3. First-launch cache is always cold

The first time you play any game, every shader must be compiled from scratch. No DXVK build, including VEGAS, can avoid this. Async hides the stalls (the game keeps rendering while compilation happens in the background), but you will see missing shaders appear as black/invisible geometry or flickering until compilation completes. This is not stutter — it's async working correctly. The shaders get cached for subsequent launches.

VEGAS's 60-second cache flush does help here: even if the emulator is killed mid-game, the shaders compiled so far are saved to disk. Stock DXVK loses them all on unclean shutdown, forcing a full recompile next time.

4. Turnip driver compilation is slow on low-end Adreno

On Tier 1 GPUs (Adreno 610/618/619), the Turnip shader compiler itself is slow — it's running on the same limited CPU and GPU resources as the game. Even with async offloading compilation to worker threads, those threads compete with the game for CPU time. The 4-thread cap in VEGAS helps significantly, but on a 6-series Adreno with Cortex-A76 cores, compilation of complex shaders can take 50-100ms per pipeline. If 10 new shaders hit in one frame, that's 500ms+ of compilation — even async can't hide all of it.

5. Pipeline cache is corrupted or not persisting

If stutter is equally bad on the 2nd, 3rd, and 10th launch, your pipeline cache may not be persisting. Common causes:

  • The emulator's container storage is read-only or gets wiped on each launch
  • The Wine prefix is configured to use a temporary directory
  • Multiple app instances conflict on the same cache files

Check if the DXVK state cache file (*.dxvk-cache) grows between launches. If it resets to 0 bytes each time, the cache isn't persisting.

6. The "90% reduction" claim is an average, not a guarantee

In real testing:

  • Well-optimized games (Tomb Raider 2013, Prey 2017): stutter reduced by 90-95% with async + GPL fix + thread cap
  • Shader-heavy games (Unity titles, Unreal Engine 4): stutter reduced by 60-80% — more shaders means more unavoidable compilation
  • Worst case (games that compile shaders at specific trigger points like entering a new area): stutter reduced by 40-50% — the compilation spike is concentrated and even async can't fully hide a sudden burst of 50+ pipelines
Bottom line: If you're still stuttering hard, check these in order: (1) is dxvk.enableAsync = True set? (2) is your pipeline cache persisting between launches? (3) have you given the game time to warm up its shader cache on the first play session? VEGAS reduces stutter by 90% for the average case, but the first playthrough of a shader-heavy game will always have some hitches.
The game crashes or doesn't launch — what should I try?

Try these steps in order:

  1. Disable VEGAS entirely — Set dxvk.enableStarProfile = False in your config. If the game launches with stock DXVK behavior, the issue is VEGAS-specific.
  2. Disable async FSR — Set vegas.enableUpscaler = False.
  3. Disable frame generation — Frame gen is off by default on Tier 1 and conditional on Tiers 2/3, but force it off: dxvk.enableAsync = False (also disables GPL).
  4. Force a lower tier — vegas.forceTier = 1 — this disables frame gen and uses conservative thresholds.
  5. Reduce compiler threads — dxvk.numCompilerThreads = 2.
  6. Check the log — Enable DXVK_LOG_LEVEL=debug and look for "error", "VUID-", or "Vegas:" lines in logcat.
  7. Try a different Turnip driver version — Mesa 25.x is recommended. Some games need specific driver builds.
If the game still crashes with dxvk.enableStarProfile = False, the issue is not VEGAS-specific — it's a DXVK or driver compatibility issue with that game.
Can I use VEGAS with Wine / GameNative?

Yes. VEGAS is a drop-in replacement for DXVK. It works with any Wine-based Android emulation layer, including:

  • Star Emulator — Native support via VEGAS-type WCP packages.
  • Winlator — Install via DXVK-type WCP package or manual DLL replacement.
  • GameNative — Same Winlator-compatible approach. VEGAS uses the same Wine + DXVK stack.
  • Any Wine wrapper on Android — Replace the DXVK DLLs in the Wine prefix.

VEGAS does not modify Wine's own behavior — it only changes the D3D→Vulkan translation layer. All Wine settings (audio, input, window management) remain unchanged.

How do I build VEGAS from source?

VEGAS is built using MinGW cross-compilation on Linux. The recommended approach is via GitHub Actions, but you can also build locally:

git clone --recursive https://github.com/isygold/Vegas-Private.git
cd Vegas-Private

# Install dependencies (Fedora)
sudo dnf install mingw64-gcc mingw64-gcc-c++ mingw64-winpthreads \
  mingw64-vulkan-headers glslang meson ninja-build

# Build
./package-release.sh 2.7.3-vegas build --no-package --64-only --build-id

The output DLLs are placed in build/dxvk-2.7.3-vegas/x64/. For the WCP package, see the wcpbuild.yml workflow in the repo.

Android cross-compilation (NDK) is not currently the primary build target. The MinGW cross-build produces Windows DLLs that run under Wine on Android. This is standard for all DXVK forks on Android.
What is the BCn→ASTC transcoder?

The BCn→ASTC transcoder is a two-pass GPU compute pipeline in VEGAS that converts BCn-compressed textures (DXT1/3/5, BC4/5/7) to ASTC 4×4 format before upload to the GPU.

Two-pass pipeline:

  1. BCn decode — Decodes BC1–BC7 to RGBA8 scratch buffer
  2. ASTC encode — leegao's PCA-based ASTC 4×4 encoder (optimized for quality)

Status: ⚠️ Development / Gated. The code is implemented but not wired into the active upload pipeline by default. The simplified ASTC encoder produces visual quality loss that outweighs the narrow benefit (only helps old Qualcomm blob drivers, not Turnip).

Future developers interested in completing this feature should properly wire it into the staging-buffer upload path and verify block-size alignment (BCn is always 4×4 blocks, ASTC uses 5×5 or 6×6 blocks — non-power-of-2 texture dimensions are a known risk).

VEGAS vs DXVK 3.0 — should I upgrade?

No, not for Android. DXVK 3.0 (released June 25, 2026) requires Vulkan 1.4. The Turnip driver on Adreno currently supports Vulkan 1.3 at best. DXVK 3.0 will not work on any Android Adreno device.

VEGAS is based on GPLAsync DXVK 2.7.1 and DXVK 2.7.1(Vulkan 1.3). Key differences:

FeatureVEGAS 2.7.3DXVK 3.0
Vulkan required1.31.4
Android / Adreno✅ Fully supported❌ Unusable (no Vulkan 1.4 driver)
TBDR optimizations✅ Extensive❌ None (desktop focused)
FSR upscaler✅ Built-in❌ None
Frame generation✅ Built-in❌ None
Async pipeline compilation✅ GPLAsync compatible❌ Removed

If someone adds VK_KHR_load_store_op_none to VEGAS (cherry-pickable from DXVK 2.6.1+), Adreno performance would further improve by avoiding unnecessary tile-buffer flushes on temporary attachments.

How do I report a bug?

Bugs and feature requests are tracked on the Vegas-Private repository issues page. When reporting a bug, please include:

  1. Device info: Adreno model, Turnip driver version, Android version
  2. Game info: Title, engine (if known), D3D version (9/10/11)
  3. Config: Your full dxvk.conf (with any sensitive info removed)
  4. Logs: Full DXVK logcat output with DXVK_LOG_LEVEL=debug
  5. Reproduction steps: What you did, what happened, what you expected
  6. VEGAS version: Release tag or commit SHA

Before reporting, try these first:

  • Test with dxvk.enableStarProfile = False to confirm it's a VEGAS-specific issue
  • Search existing issues for similar reports
  • Try different config options (force tier, disable features) to narrow down the cause
The most helpful bug reports include a before/after comparison with the master switch toggled off. This instantly tells the developer whether the bug is in VEGAS code or in the base DXVK.
What is the GPLAsync build and how does it relate to VEGAS?

GPLAsync is a community fork of DXVK that adds async pipeline compilation on top of the Graphics Pipeline Libraries (VK_EXT_graphics_pipeline_library) extension. It was the first DXVK variant to successfully combine GPL with dxvk.enableAsync without race conditions — a problem that stock DXVK and earlier async builds could not solve.

VEGAS traces its lineage through three layers:

LayerBaseKey Addition
DXVK (upstream)v2.7.1Direct3D 9/10/11 → Vulkan translation. The foundation by doitsujin.
GPLAsync forkDXVK base 2.7.1Async pipeline compilation with GPL compatibility. Fixed the GPL-vs-async race that plagued all earlier attempts.
VEGASGPLAsync 2.7.1 buildAll of the above + TBDR governor, FSR 1.0, frame generation, tier tuning, Adreno-specific driver workarounds.

What VEGAS inherited from GPLAsync:

  • The core dxvk.enableAsync mechanism that offloads shader compilation to worker threads
  • GPL detection + automatic disable when async is active (prevents the race condition)
  • The pipeline compilation worker thread infrastructure

What VEGAS improved on top of GPLAsync:

  • Compiler thread cap — GPLAsync used hardware_concurrency() which spawns 8 threads on a typical big.LITTLE SoC. VEGAS caps this at 4 on ARM64 (see FAQ #3).
  • Shader cache persistence — VEGAS adds the 60-second cache flush so partially-compiled caches survive emulator kills (see FAQ #3).
  • All TBDR features — GPLAsync is desktop-focused. VEGAS adds the Adreno-specific draw governor, FSR, frame gen, tier system, etc.

The dxbc-spirv connection: Both the base DXVK and GPLAsync use dxbc-spirv as the shader compiler backend. This is the subproject in subprojects/dxbc-spirv/ that translates DXBC bytecode (from D3D games) into SPIR-V (for Vulkan). VEGAS inherits this unchanged — it does not modify the shader compiler. The dxbc-spirv path is the same across VEGAS, GPLAsync, and upstream DXVK. The only difference is in how the compiled pipelines are submitted to the GPU (async vs synchronous).

Why this matters: If you've used a GPLAsync build before, VEGAS will feel familiar but with dramatically better performance on Adreno. The async compilation you relied on is still there, and VEGAS fixes the remaining issues (thread oversubscription, cache loss on kill) that GPLAsync left unaddressed on mobile.
